# Lambda (v1.1) — Design

## Context

v1's architecture doc (`2026-05-12-keyten-cli-architecture.md:116`) defers
user-defined functions to v1.1. v2 stage 2 added data-parallel kernel
execution and verified the substrate (atomic refcount, `RefObj: Send`,
chunk-step parallelism, TSan-clean). The next planned chapter — **v3
reactor** (reagents + reactions, join-calculus atomic multi-channel
commits) — needs first-class lambdas because **reactions ARE lambdas**
with reagent-binding metadata. Without lambdas, there's nothing for a
reaction to fire.

So v1.1 is now on the critical path:

- v2 (data-parallel kernels) ✅
- **v1.1 (lambdas) — this doc**
- v3 (reactor) — blocked on v1.1

Lambdas are also independently useful: K-style `{x+1}` syntax for
ad-hoc functions, named functions via assignment (`f: {…}`), and the
foundation for higher-order operations like `each` (`{f x}'list`),
generic adverbs beyond `+/`, and conditionals (`$[…]`).

## Intended outcome

A keyten user can:

```
𝕜 f: {x*x}
𝕜 f[7]
49
𝕜 {[x;y] x+y}[3; 4]
7
𝕜 sq: {x*x}; +/sq'!10
285        / sum of squares 0..9 once `'` (each) lands; v1.2 work
```

Reactions can then layer on top:

```
𝕜 reaction[c1; c2; {[a;b] a+b}]   / v3 reactor — depends on this doc
```

## Scope (v1.1) — design decisions resolved 2026-05-12

| In | Out (v1.2+) |
|---|---|
| `{body}` with **implicit `x`, `y`, `z` auto-detected** (K convention: arity = max position of any free `x`/`y`/`z` referenced in the body) | Curried partial application |
| `{[x;y] body}` (explicit param list, 0–8 args, `;`-separated per K convention) | `each` (`'`), `over` for user-defined verbs |
| **Bracket-apply `f[a;b;c]`** and **juxtaposition `f x`** (K-idiomatic monadic form) | Lexical closures with env capture |
| Lambdas as first-class values (assign, pass, return) | Tail-call optimization |
| **Push new env frame per call** — params bound in a fresh inner scope; outer names looked up via scope chain | Mutual recursion validation |
| Recursion via global name (`f: {…f[…]…}`) — outer scope sees the assignment | |
| Sync + async eval through existing `eval_async` | |

### Implicit-arg rules (K convention)

When a `{body}` has no explicit `[params]` list, the parser walks the
body to detect which of the free variables `x`, `y`, `z` appear. The
inferred arity is the **maximum position** of any seen variable:

| Body uses | Inferred arity | Inferred `params` |
|---|---|---|
| nothing | 0 | `[]` |
| only `x` | 1 | `[x]` |
| `y` (with or without `x`) | 2 | `[x, y]` |
| `z` (with or without `x`/`y`) | 3 | `[x, y, z]` |

So `{y}` is dyadic; calling it as `{y}[3;7]` returns `7` (x discarded).
This matches K idiom precisely.

### Juxtaposition apply rules

K's grammar is noun-verb-noun for dyadic, verb-noun for monadic. For
user lambdas treated as nouns-holding-callable-values:

- `f x` → `f[x]` (monadic apply via juxtaposition)
- `f x y` → `f[x; y]`? **No** — K reads right-to-left: `f (x y)`. So
  multi-arg juxtaposition isn't supported; bracket form is required for
  arity > 1.
- `x f y` → `f[x; y]` — name in dyadic position is dyadic. **Not in
  v1.1**: requires verb-noun arity disambiguation and a name-resolution
  query at parse time (or runtime). Deferred to v1.2.

The v1.1 juxtaposition rule reduces to: `primary ws primary` →
`Apply { func = first, args = [second] }`. The trailing primary
absorbs nothing else (no left-associative `f x y z` cascades).

## AST changes

```rust
// keyten/src/ast.rs
pub enum Expr {
    // … existing variants unchanged …

    /// `{[x;y] body}` or `{body}`. `params` is the explicit parameter list;
    /// empty `params` denotes the implicit-x form (single arg named `x`).
    Lambda {
        params: Vec<Sym>,
        body: Box<Expr>,
        span: Span,
    },

    /// `f[a;b;c]` — application of a function value to `n` arguments.
    /// `func` evaluates to a `Kind::Lambda` RefObj; `args` is empty for
    /// nullary `f[]`.
    Apply {
        func: Box<Expr>,
        args: Vec<Expr>,
        span: Span,
    },
}
```

`Apply` only handles the bracket form `f[args]` in v1.1. Juxtaposition
`f x` requires non-trivial parser changes (noun/verb arity disambiguation)
and is deferred.

## Heap representation

Lambdas are first-class values, so they need a heap representation
reachable from `RefObj`. Design:

- New variant `Kind::Lambda` with code `-30` (atom-shaped — single value,
  not a vector).
- Atom-style 16-byte cell from the existing per-thread freelist
  (`runtime.rs:32`).
- Payload (offset 8, 8 bytes): a `*mut LambdaInner` pointer.
- `LambdaInner` is `Box`-allocated outside the Obj cell, holds params +
  body. `Arc<Expr>` for the body so cloning a lambda is cheap.

```rust
// keyten/src/alloc/lambda.rs (or extension to alloc.rs)
pub struct LambdaInner {
    pub params: Vec<Sym>,
    pub body: Arc<Expr>,
}

pub unsafe fn alloc_lambda(inner: LambdaInner) -> RefObj {
    let p = RUNTIME.pop_atom_cell();
    write_header(p, /*meta=*/0, /*attr=*/0,
                  Kind::Lambda.atom(), /*rc=*/1);
    let boxed = Box::into_raw(Box::new(inner));
    ptr::write_unaligned((p as *mut u8).add(8) as *mut *mut LambdaInner, boxed);
    RefObj(NonNull::new_unchecked(p))
}
```

### Release path

`release()` in `alloc.rs:102-138` currently routes atoms straight to the
freelist push. With Lambda, the boxed inner must be dropped first:

```rust
if raw < 0 {
    // Atom path
    if raw == Kind::Lambda.atom() {
        let inner_ptr = ptr::read_unaligned(
            (p.as_ptr() as *const u8).add(8) as *const *mut LambdaInner
        );
        drop(Box::from_raw(inner_ptr));
    }
    RUNTIME.push_atom_cell(p.as_ptr());
    return;
}
```

This is the only special case in the release routing — the freelist push
itself is unaffected (the cell's bytes are reused without caring about
their prior contents).

### Why not store params + body inline in the Obj cell?

Two reasons:

1. **Cell size is fixed at 16 bytes** for the atom freelist. A
   `Vec<Sym>` is 24 bytes alone. Variable-sized payloads would break the
   freelist invariant or require a different allocation path.
2. **AST tree is heap-tree-shaped already** (`Box<Expr>`); putting an
   `Arc<Expr>` pointer in the 8-byte payload keeps the rest of the
   AST under standard Rust ownership without reinventing it.

The cost: an extra heap allocation per lambda construction. Lambdas
should not be a hot allocation path (they're constructed at parse time
or at most once per `Expr::Lambda` evaluation), so this is fine.

## Parser changes

### Tokens

Add to `parse/lex.rs`:
- `LBrace` `{`
- `RBrace` `}`
- `LBracket` `[`
- `RBracket` `]`

The bracket tokens are also needed for `Apply` (and for explicit param
lists). They were probably already in the lexer for list literals; verify.

### Grammar (`parse/grammar.rs`)

```text
lambda      := "{" param_list? body "}"
param_list  := "[" name ("," name)* "]"   # or ";" separator — pick one
body        := expr_seq                    # same as top-level
apply       := atom "[" args? "]"
args        := expr (";" expr)*
```

Apply is parsed as a postfix on any atom-level expression:

```text
atom_postfix := atom ( "[" args? "]" )*
```

This lets `f[1][2]` work (apply twice), useful once curried application
lands in v1.2.

For implicit `x`: when a `{body}` has no explicit `[params]` list, the
parser injects `params = vec![sym!("x")]`. The body may freely reference
`x`; it's looked up in the call-time env (since v1.1 has no closures,
this is just the normal env binding).

### Precedence

Apply binds tighter than any binary verb. `f[3] + 5` is `(f[3]) + 5`.
Lambda construction `{…}` is a primary expression at lowest precedence
inside its braces, just like `(…)` grouping is.

## Eval changes

The chosen scoping discipline (**push new env frame per call**) replaces
v1's flat `Env: HashMap<Sym, RefObj>` with a scope-chain structure.
The simplest representation that supports v1.1's needs:

```rust
// keyten/src/eval/env.rs
pub struct Env {
    /// Stack of scopes, innermost at the back. The outermost is the
    /// global REPL env. Assignments via `:` target the **outermost**
    /// scope (so functions can define top-level names). Lambda calls
    /// push a fresh scope for their params and pop on return.
    frames: Vec<HashMap<Sym, RefObj>>,
}

impl Env {
    pub fn new() -> Self { Self { frames: vec![HashMap::new()] } }
    pub fn push_frame(&mut self) { self.frames.push(HashMap::new()); }
    pub fn pop_frame(&mut self) { self.frames.pop(); /* leave at least global */ }
    pub fn bind_local(&mut self, sym: Sym, v: RefObj) {
        self.frames.last_mut().unwrap().insert(sym, v);
    }
    pub fn bind_global(&mut self, sym: Sym, v: RefObj) {
        self.frames.first_mut().unwrap().insert(sym, v);
    }
    pub fn lookup(&self, sym: Sym) -> Option<RefObj> {
        // Search inner → outer.
        for frame in self.frames.iter().rev() {
            if let Some(v) = frame.get(&sym) { return Some(v.clone()); }
        }
        None
    }
}
```

`Expr::Assign` continues to call `bind_global` (top-level assignment
semantics are unchanged for v1 callers; assignments inside lambda
bodies via the standard `x: 5` syntax target the global env).
Future v1.2 work could add a `local` form for explicit shadowing if
needed.

Apply evaluation:

```rust
Expr::Lambda { params, body, .. } => {
    let inner = LambdaInner {
        params: params.clone(),
        body: Arc::new((**body).clone()),
    };
    Ok(unsafe { alloc_lambda(inner) })
}

Expr::Apply { func, args, span } => {
    let f = eval_boxed(func, env, ctx).await?;
    if f.kind() != Kind::Lambda {
        return Err(EvalErr::NotCallable { kind: f.kind(), span: *span });
    }

    let mut arg_values = Vec::with_capacity(args.len());
    for a in args {
        arg_values.push(eval_boxed(a, env, ctx).await?);
    }

    let inner_ptr: *mut LambdaInner = unsafe { f.atom() };
    let inner: &LambdaInner = unsafe { &*inner_ptr };

    if arg_values.len() != inner.params.len() {
        return Err(EvalErr::Arity {
            expected: inner.params.len(),
            got: arg_values.len(),
            span: *span,
        });
    }

    env.push_frame();
    for (p, v) in inner.params.iter().zip(arg_values.into_iter()) {
        env.bind_local(*p, v);
    }
    let result = eval_boxed(&inner.body, env, ctx).await;
    env.pop_frame();
    result
}
```

The frame push/pop pattern means params shadow any outer binding for
the duration of the call without explicit save-and-restore — cleaner
than the v1.1 MVP design that I'd initially sketched. Recursion still
works because `f` is resolved through outer scopes (the global env)
even during the inner call.

## Tests (acceptance gates)

```
{x}[5]                            → 5            // identity
{x+1}[10]                         → 11           // monadic
{[x;y] x+y}[3;4]                  → 7            // dyadic
{[x;y;z] x+y+z}[1;2;3]            → 6            // triadic
f: {x*x}; f[7]                    → 49           // named, reuse
f: {[n] n + 1}; f[f[f[0]]]        → 3            // nested apply
+/{x*x}'!10                       // deferred — needs `'` (each), v1.2
```

Recursive test (relies on global name lookup):

```
fac: {[n] $[n<2; 1; n*fac[n-1]]}; fac[5]   → 120
```

This requires conditional `$[…]` which is itself deferred. For v1.1
acceptance, the recursive test stays in the deferred bucket. We can
substitute a non-recursive test that captures the binding behaviour:

```
double: {x*2}; double[double[3]]            → 12
swap: {[x;y] (y;x)}; swap[1;2]              → (2;1) // deferred — list literal
```

## Commit sequence (estimated 8–10)

The decision to include juxtaposition apply + implicit `x`/`y`/`z`
detection adds two commits over the MVP I'd recommended. Sequence:

1. **`env` scope-chain refactor**: replace `Env: HashMap<Sym, RefObj>`
   with the `frames: Vec<HashMap<…>>` design above. All v1/v2 tests
   must continue to pass (single-frame mode == v1 behavior). ~80 LOC,
   pure refactor.
2. **AST**: add `Lambda` and `Apply` variants + `Span` accessors. ~50 LOC.
3. **Lex**: add `{`, `}`, `[`, `]` token variants if not already present
   for list literals. Verify against existing tokenizer.
4. **Parse**: lambda construction + bracket-apply postfix parser. The
   juxtaposition apply rule (primary followed by primary) requires a
   small grammar refactor to recognise adjacent atoms as application.
   ~150 LOC.
5. **Implicit-param free-variable analysis**: a small visitor over the
   lambda body to detect free references to `x`/`y`/`z`; arity = max
   seen position. ~40 LOC.
6. **Kind / alloc**: `Kind::Lambda`, `alloc_lambda`, release-path
   special-case (drop `Box<LambdaInner>` before freelist push). ~60 LOC.
7. **Eval**: `Lambda` + `Apply` arms; new `EvalErr::NotCallable` and
   `EvalErr::Arity`. ~80 LOC.
8. **End-to-end tests** in `tests/parse_eval.rs` (the acceptance gates
   above). ~100 LOC.
9. **TSan run** to confirm no new races from the lambda allocator path.
10. Optional: REPL syntax highlighting for `{}` and `[]`
    (`keyten-cli/src/highlighter.rs`).

Estimated total: ~600 LOC, ~3-5 focused sessions.

## Verification

End-to-end:
1. `cargo test -p keyten` — all v1 + v2 tests still pass.
2. New `parse_eval` tests for the listed acceptance gates pass.
3. `cargo run -p keyten-cli --release` — REPL parses and evaluates the
   acceptance examples interactively.
4. `cargo bench -p keyten --bench bench_rc` — no regression vs. the
   atomic-rc baseline (lambda construction shouldn't touch rc hot
   path, but the lambda kind is atom-allocated through the same
   freelist, so confirm).
5. TSan run (`RUSTFLAGS="-Z sanitizer=thread" cargo +nightly test ...`)
   stays clean — lambda construction allocates a Box and stores its
   pointer; this is a single non-atomic operation, but the `RefObj`
   wrapping it is properly atomic-refcounted, so the lambda is safe
   to share across threads as long as the body is.

## What v1.1 unblocks

- v3 reactor: reactions are lambdas with reagent-binding metadata.
- v1.2 each-adverb (`'`): `f'list` evaluates `f x` for each element.
- v1.2 conditionals: `$[cond;t;e]` is best implemented as a builtin
  that takes lambdas for the branches.
- General higher-order operators that need callable values.

## Out of scope (deferred to v1.2+, v2.1, v3)

- Lexical closures.
- Juxtaposition apply.
- Tail-call optimization.
- Mutual recursion validation.
- Generic verbs (named binary functions usable in `over`, `each`).
- Parser improvements for K-style infix application (`x f y`).
